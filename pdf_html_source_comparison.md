# HTML 与 PDF 完整规格输入源对比

前提：HTML 和 PDF 的业务信息一致，PDF 是从 HTML 转换而来。因此二者的区别不在规格信息内容，而在机器解析时可保留多少结构信息。

| 维度 | HTML 完整规格 | PDF 完整规格 |
|---|---|---|
| 信息内容 | 与 PDF 一致，是完整规格信息来源 | 与 HTML 一致，是 HTML 转换后的版式文件 |
| 关系 | 上游结构化源 | 下游展示/分发格式 |
| 信息完整性 | 不应比 PDF 多，也不应少 | 不应比 HTML 多，也不应少 |
| 主要区别 | 保留 DOM 结构、标签层级、表格单元格 | 保留视觉版式、分页、排版位置 |
| 机器解析难度 | 较低，能直接定位 `spec-content`、产品名、标题、表格、列表 | 较高，需要从页面版式中反推出文本顺序和表格结构 |
| 产品名获取 | 可直接从 `titleProductName` 获取 | 通常依赖文件名或文本抽取结果 |
| Feature 边界 | `h1 / h2 / h3 / divFeatureValue` 边界明确 | 依赖抽取文本中的标题和换行判断 |
| 表格结构 | `tr / td` 顺序明确 | 表格可能被抽成多行、多列、跨页片段 |
| 跨页问题 | 不存在物理分页导致的字段粘连 | 存在分页、页眉页脚、跨页表格带来的解析风险 |
| 解析错误来源 | 主要来自 HTML 结构变化或转换层规则不足 | 主要来自 PDF 文本抽取、分页、列顺序、版式还原 |
| 规则复用 | 可以转换成现有 spec text 后复用 Storage/WLAN/Display 规则 | 当前规则已直接基于 PDF 抽取文本运行 |
| 可审计性 | 更容易追踪字段属于哪个规格项 | 需要结合抽取文本和 PDF 视觉位置判断 |
| 稳定性 | 对程序更稳定，因为结构信息还在 | 对人阅读稳定，对程序解析不如 HTML 稳定 |
| 适合作为长期输入 | 更适合自动化生成 | 适合人工查看和历史兼容 |

更准确的结论是：

HTML 和 PDF 的规格信息是一致的；HTML 的优势不是信息更多，而是结构更完整、更适合程序解析。PDF 的问题不是规格内容不完整，而是从 PDF 反抽文本时可能丢失 HTML 原本拥有的结构信息，例如表格单元格、字段边界和分页前后的关系。

因此，如果两者都可获得，自动生成工具应优先使用 HTML；PDF 更适合作为人工确认、历史兼容或 HTML 不可用时的输入。
